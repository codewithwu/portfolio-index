RuleEngine 规则引擎 核心

基于正则匹配的快速意图识别与参数提取引擎，无需LLM即可解析用户查询

  • match(query) → 返回 (意图, 参数)
  • _extract_price() 提取价格区间
  • 支持品牌、车型、颜色、销售员识别
  • 两种意图：query_car / query_salesperson
  • build_response() 构建查询响应
匹配示例
"查询丰田轿车" → query_car, {brand:丰田, model:轿车}
"红色SUV有哪些" → query_car, {model:SUV, color:红色}
"不超过6万的车" → query_car, {max_price:60000}
"哪个销售员卖得最多" → query_salesperson

rule_based_query 工具 NLU

自然语言理解风格的查询工具，自动从用户输入中识别意图和参数

  • 输入："张三卖了多少辆车"
  • 内部调用 RuleEngine.match()
  • 自动解析品牌/车型/价格等参数
  • 无需手动指定查询字段
  • 返回格式化结果字符串
调用链
用户: "查询丰田轿车的价格"
→ rule_based_query.invoke()
→ engine.match() → query_car
→ engine._query_car(df, params)
🔍
query_car_data 工具 语义

传统的多参数组合查询工具，需要LLM理解并填充参数

  • 参数：brand, model, color
  • 参数：sales_person
  • 参数：min_price, max_price
  • 参数：limit (默认10条)
  • 直接对DataFrame进行条件筛选
参数示例
brand="丰田", model="轿车"
color="红色", max_price=60000
sales_person="张三", limit=10
📦
State 定义 — MessagesState + 自定义字段 状态

继承自 LangGraph 的 MessagesState

  • messages: 对话消息历史
  • 自动支持多轮对话
  • LLM调用结果追加到此

自定义状态字段

  • is_true: bool = False
  • 标记规则引擎是否匹配成功
  • 控制条件路由的分支选择
⚖️
两种方案对比

规则引擎方案

  • ✅ 快速 — 无需LLM调用
  • ✅ 确定性 — 正则匹配，结果可预期
  • ✅ 低成本 — 节省API费用
  • ❌ 覆盖有限，需要预定义模式
  • ❌ 复杂查询难以处理

LLM + Tools方案

  • ✅ 灵活 — 自然语言理解
  • ✅ 泛化 — 处理未见过的问题
  • ✅ 智能 — 复杂推理能力
  • ❌ 较慢 — 需要LLM推理
  • ❌ 成本高 — API调用费用
